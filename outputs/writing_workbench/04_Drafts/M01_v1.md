## 3. 金融问题表达

### 3.1 核心问题

**哪些研发资本化特征与交易所问询相关？**

这一问题的金融直觉来自对 A 股研发密集型企业的长期观察：研发资本化涉及大量会计估计（技术可行性、未来经济利益可流入、开发支出可计量），管理层天然有动机通过资本化率调节利润。当一家公司的资本化行为与同行业显著偏离、或资本化率在短期内剧烈跳升时，这种"尾巴风险"是否更容易触发交易所问询？

### 3.2 三个可检验的子问题

| 子问题 | 分析维度 | 数据支撑 |
| --- | --- | --- |
| 子问题 1：高资本化率是否更容易与问询相关？ | 资本化率水平 vs 行业偏离度 | 150 份年报 |
| 子问题 2：资本化率的剧烈变化是否更容易与问询相关？ | 跨期变化幅度（2021→2022→2023） | 150 份年报 |
| 子问题 3：资本化条件描述模糊是否更容易与问询相关？ | 条件文本模糊度评分 | 150 份年报 + 问询标签 |

### 3.3 为什么这不是一个"违规判定"问题

需要从一开始就明确：**本项目做的是风险排序，不是违规判定**。模型输出的是一个供审计师/分析师复核的排序清单——"这些公司的资本化行为在统计上最偏离同行，建议优先复核"。模型不会、也不应该声称"某家公司违规了"。这一区分贯穿了本项目的所有设计和措辞。

## 4. 研究设计与方法链路

### 4.1 观察单位与变量体系

观察单位是 **company-year**（公司 $i$ 在年报年度 $t$ 的研发资本化记录）。

核心因变量 `capitalization_related`：该 company-year 是否收到实质针对研发资本化的交易所问询函。自变量包括资本化率（`capitalization_rate`）、行业百分位（`industry_percentile`）、跨期变化 Z-score（`change_zscore`）、条件文本模糊度（`fuzziness_score`），以及恒等式置信度乘数（`identity_check_score`）。

### 4.2 数据范围

| 维度 | 设计 | 与课程要求的对应 |
| --- | --- | --- |
| 公司池 | 50 家（医药制造 20 + 电子设备 20 + 软件信息 10） | 覆盖研发密集型行业 |
| 时间范围 | 2021、2022、2023 年 | 支持跨期趋势分析 |

### 4.3 技术链路（全流程 13 阶段）

```text
crawl → download → audit → parse → route → extract → validate → score → detect → inquiry → inquiry-download → inquiry-label → analyze → report
```

1. **crawl**：官方 API（OAuth2）抓取年报 metadata，150/150 成功
2. **download**：PDF 下载，150/150，collision=0（SHA256 唯一性验证通过）
3. **audit**：8 项数据质量检查全部通过
4. **parse**：MinerU API batch 将 150 份 PDF 转为 Markdown，275 个 page segment 全部 success
5. **route**：基于关键词规则定位"研发投入""开发支出"等章节，输出 750 行 section 检查报告
6. **extract**：确定性 HTML 表格提取 + LLM 抽取双路线，每个关键字段带 `evidence_text`
7. **validate**：Pydantic 校验 + 会计恒等式（资本化 + 费用化 ≈ 总额，5% 阈值），150/150 passed
8. **score**：四维风险评分，83 条有可用风险分，67 条因关键字段缺失未评分
9. **detect**：按风险分前 20% 标记异常，17 条异常（industry_outlier 16 / change_spike 1）
10. **inquiry**：全样本问询候选发现，28 条候选
11. **inquiry-label**：v2 语义标签（Tier-1 关键词剪枝 + LLM 语义二分类），1 条 related
12. **analyze**：混淆矩阵计算 + baseline 对比 + TN 水分分析
13. **report**：自动技术底稿写入 `outputs/final_report_auto.md`

### 4.4 关键设计决策

- **不做时间序列回归**：150 条记录不足以支撑复杂的因果推断，改为横截面比较（每年每个行业的偏离度）。
- **风险评分不是分类器**：没有训练任何 ML 模型，全部使用可解释的金融规则。
- **问询标签不是真值**：问询函相关性标签基于 Tier-1 关键词剪枝 + LLM 语义二分类，标注为 MVP 版本，不是监管级判定。

### 5.3 Section 路由

基于 `configs/section_rules.yaml` 的 5 条规则（研发费用、开发支出、无形资产、资本化会计政策、收入确认），对 150 份年报的 Markdown 进行关键词匹配 + 正负向打分，产出 section 检查报告（`outputs/reports/section_check_report.csv`）。

### 5.4 字段抽取

采用**双重路线**：确定性 HTML 表格提取器优先（从 MinerU Markdown 的 `<table>` 标签直接解析研发投入/开发支出数值），LLM 抽取做补充（对表格提取无法覆盖的资本化条件文本等自由文本字段）。每个关键字段携带 `evidence_text`（原文片段），确保可追溯。

### 5.5 数据质量校验

Pydantic 校验 + 会计恒等式交叉验证（5% 阈值）：

```text
diff_ratio = abs(capitalized + expensed - total) / total
若 diff_ratio > 5% → 标记为恒等式不一致
```

150 条记录全部通过 validate 阶段。

## 6. 风险评分模型

### 6.1 金融逻辑

风险评分模型基于一个核心直觉：**当公司的研发资本化行为在多个维度上同时表现出"异于同行"的特征时，这家公司的会计政策选择就更值得关注**。四个维度分别度量了水平偏离、变化幅度、条件模糊度和数据可信度。

### 6.2 四维公式

三个正向信号权重均为 0.25：

| 维度 | 公式 | 含义 |
| --- | --- | --- |
| 行业偏离度 | $IS = P_{i,t} \times 100$ | 本公司资本化率在同行业同年度的百分位 |
| 跨期变化 | $CS = \min(100, \frac{\max(0, Z)}{2} \times 100)$ | 只取正向跳升（资本化率突然上升），不惩罚下降 |
| 条件模糊度 | $FC = \min(1, \frac{FH}{\max(1, len/200)}) \times 100$ | 模糊关键词命中数做长度归一化 |

恒等式置信度乘数：

$$I = \max(0, 1 - \frac{diff\\_ratio}{0.20})$$

综合风险分：

$$AS = \frac{IS + CS + FC}{3} \times I$$

取值范围 $[0, 100]$。注意 $I$ 是乘数而非加项——当金额关系不一致时，降低整体风险分的置信度，不直接解读为公司更激进。这体现了"不确定时不妄下结论"的审慎原则。

### 6.3 参数与待审计项

| 参数 | 当前值 | 说明 |
| --- | --- | --- |
| 三维权重 | 各 0.25 | 等权，缺乏实证校准 |
| 异常分位 | 前 20% | 相对阈值 |
| change_zscore 缩放因子 | 2 | 控制变化评分范围 |
| 模糊度归一化基准 | 每 200 字符 | 避免长文本被高估 |
| 恒等式折扣基准 | 20% | 相对偏差 20% 时折扣为 0 |

所有参数均为启发式设定，未经过系统敏感性分析——这是 §13 局限性的核心内容之一。

### 8.2 TN 水分说明

**TN=132 并非 132 条记录都通过了模型判定**。其中 67 条记录因关键字段缺失（缺少资本化金额或费用化金额）无法计算风险分，在流转到 detect 阶段时被默认归为非异常（TN）。这 67 条相当于"未经过模型检验就被放行"的记录，TN 因此被系统性高估。

扣除这 67 条水分后，真正有风险分且被判定为非异常的记录为 132 − 67 = 65 条。

### 8.3 唯一正样本（FN：汇顶科技 2022）

150 条记录中，仅 **汇顶科技（603160）2022 年报** 被问询标签判定为实质针对研发资本化的监管函件：

- 监管函件：关于对"关于深圳市汇顶科技股份有限公司2022年年度报告的信息披露监管工作函"的回复公告
- 问询关键词：无形资产、研发、研发费用
- LLM 语义确认：confidence=0.9
- 模型风险分：0.0（资本化率字段缺失）
- 预测结果：**FN**（漏报）

**直接原因**：该记录的 `capitalization_rate` 因字段缺失无法计算，导致所有评分分量均为 0/缺失，模型无法对其排序。这不是评分逻辑的问题，而是**输入数据覆盖不足**的问题——模型的设计前提是"有字段才能评分"，但恰好唯一的正样本存在于字段缺失的集合中。

### 8.4 Baseline 对比

为诚实评估规则评分是否比简单排序提供了增量信息，我们设置了三条 baseline：

| Baseline | 预测异常数 | TP | FP | FN | Precision | Recall |
| --- | --- | --- | --- | --- | --- | --- |
| 规则评分（本项目） | 17 | 0 | 17 | 1 | 0.0 | 0.0 |
| 按资本化率排序取前 20% | 12 | 0 | 12 | 1 | 0.0 | 0.0 |
| 全标为正（全抓策略） | 150 | 1 | 149 | 0 | 0.0067 | 1.0 |

结论：在仅有 1 条正样本的条件下，**所有策略的 precision 都接近 0**。全标为正虽然 recall=1.0，但 precision=0.0067——相当于要复核 150 份年报才能找到 1 条真实问询，在实际场景中不可用。

## 10. 失败案例与评估局限

### 10.1 三个层面的局限

本项目的评估局限分为三个层面：**评分模型的局限**、**抽取质量的局限**和**问询标签的局限**。

### 10.2 评分模型局限

1. **行业百分位不分方向**：当前行业偏离度评分只度量"本公司资本化率在同行业中的排序位置"，不区分偏离方向。资本化率 0%（全部费用化）和资本化率 80%（高度资本化）在排序的两端，但都被判定为"偏离"。这导致三六零（2023，0%）、深信服（2023，0%）等公司的风险分达到满分 50 分——它们的资本化率确实是行业极端值，但偏离方向更可能是保守而非激进。

2. **参数全部启发式**：三个权重（各 0.25）、异常分位（20%）、change_zscore 阈值（1.5）、模糊度归一化基准（每 200 字符）均为人工设定，未经系统敏感性分析。

3. **跨期变化只覆盖两年**：由于部分公司某年度字段缺失，真正能计算跨期变化的记录数量有限，`change_spike` 异常类型仅检出 1 条。

### 10.3 抽取质量局限（基于 35 条人工评估）

35 条人工评估样本的准确率为 46.4%（110 个可判定字段行中 51 个正确）。错误分布在：

| 错误类型 | 数量 | 典型问题 |
| --- | --- | --- |
| section_error | 27 | 漏抽开发支出附注（期初/期末混淆） |
| data_error | 20 | 金额单位换算错误 |
| hallucination | 8 | LLM 编造了不存在的数值 |
| parse_error | 6 | capitalization_condition 占位符 |
| prompt_error | 1 | Prompt 模板未能覆盖的抽取场景 |

典型案例：
- **汇顶科技 2022**：期初/期末余额混淆导致开发支出字段抽取错误（也是唯一 FN 的背景因素之一）
- **韦尔股份 2021/2022**：研发投入金额存在幻觉值
- **capitalization_condition**：部分记录中该字段为占位符而非原文

> **诚实说明**：评估 agent 与 extractor 使用同源 LLM，存在"共错"风险——extractor 犯了错，evaluator 可能也识别不出。这是当前 LLM-based 抽取的固有问题。

### 10.4 问询标签局限

- 当前问询标签为 **MVP 版本**：先用 Tier-1 关键词剪枝，仅命中泛词时送 LLM 语义二分类。
- 正样本仅 1 条——不足以支撑任何有统计意义的 Precision/Recall/F1 评估。
- 部分公司的问询函可能遗漏（关键词覆盖面有限），部分标记为"非相关"的可能实际涉及研发资本化（LLM 判断存在假阴性）。

## 11. Vibe Coding 反思

### 11.1 AI 帮了什么

本项目全程使用 Claude Code 作为 AI 编程助手（详见 `ai_usage_statement.md`）。AI 在以下方面提供了实质性帮助：

1. **全流程代码生成**：爬虫、下载器、审计器、解析器、路由、抽取、校验、评分、检测、问询标签、评估器、报告生成器共 13 个阶段的 Python 代码均由 AI 辅助完成。
2. **Bug 诊断**：MinerU 环境问题（`fbgemm.dll` 缺失、VC++ Redistributable、torch 版本冲突）由 AI 逐层排查。
3. **文档维护**：SPEC.md、architecture.md、methodology.md、HANDOFF.md 等大量文档由 AI 起草和同步。
4. **人工评估 agent**：35 条样本的 per-field 评估由 AI agent 完成（与 extractor 同源 LLM，含共错风险，已在声明中诚实标注）。
5. **方法论审计**：fuzziness 词表去毒（排除"等/相关/未来"等高频正向词）、change_zscore 方向修正（abs → max(0,Z)）、industry 显式化等关键改进由 AI 数据审查发现。

### 11.2 AI 在哪里出错

1. **fuzziness 词表含有"等/相关/未来"**：初版词表包含"等"（汉语最高频字之一），导致几乎所有有文本的记录的模糊度评分都被无差别抬升。AI 在方法论审计中发现并修复，但暴露了"未实际看数据就设词表"的问题。
2. **change_zscore 使用 abs()**：原设计同时捕捉上升和下降，但"资本化率骤降"通常是有合理原因的（如项目完成转无形资产），不应被视为激进信号。AI 在数据审查中发现并改为只取正向跳升。
3. **行业归属靠顺序推断**：初版 scorer 没有读取 crawl.yaml 的显式 industry 配置，仅靠公司在列表中的排列顺序推断行业。AI 在审计中发现"文档声称申万一级、代码实际靠猜"的不一致，补了显式配置。
4. **文档 overclaim**：早期版本的 SPEC.md 和 README.md 使用了"异常检测""闭环验证"等措辞，超出了实际证据（TP=0）。AI 在终审 review 中逐句改为了"风险排序""可行性测试"。

### 11.3 如何验证 AI 的工作

1. **硬数字强制溯源**：报告的每一个数字都必须能在 `loop_evaluation.json` 中找到对应路径。AI 不得"自己算"任何数字——这是 CLAUDE.md 中的 Decision Rule。
2. **pytest 自动化测试**：当前 51 个测试用例覆盖了 crawl、download、audit、parse、scorer、detector、evaluator、schemas 等核心模块。
3. **mini pipeline 验证**：任何代码改动后，先用 `--limit 1` 或 `--limit 3` 跑通小微管道，确认无误后再跑全量数据。
4. **人工抽查**：35 条人工评估样本覆盖了全部 17 条异常和 9 条有问询候选的记录。
5. **"文档与代码不一致→暂停确认"**：CLAUDE.md 的 Decision Rule 强制要求，当发现文档声称和代码/数据不一致时，必须先暂停向用户确认，不得自动掩盖。

## 13. 局限性与下一步

### 13.1 当前已知局限

1. **正样本太少**（1/150 = 0.67%），任何基于 precision/recall 的评估都无法给出有统计意义的结论。
2. **TN 含水**（67/132 = 50.8%），模型的"非异常"判定有一半未经实际评分。
3. **问询标签为 MVP**，可能存在漏标（假阴性）和错标（假阳性）。
4. **page_no 缺失**，证据回溯依赖 evidence_text 原文片段而非稳定页码。
5. **35 条人工评估的准确率仅 46.4%**，抽取质量有大幅提升空间。
6. **所有评分参数为启发式设定**，未校准。
7. **评估 agent 与 extractor 同源 LLM**，共错风险未消除。
8. **行业百分位不分方向**，保守型偏离也被判为异常。

### 13.2 如果有更多时间

1. **扩大公司池**到 80-100 家，提高正样本期望数。
2. **全文语义问询标签**：用 LLM 精读每份问询函全文（而非仅标题和前几页），提高 $Y$ 标签质量。
3. **参数敏感性分析**：对异常分位、权重、change_zscore 阈值等做网格搜索，看结论是否对参数选择鲁棒。
4. **引入更多信号维度**：研发投入增速、无形资产转出占比、审计师特征等。
5. **抽取质量提升**：针对 section_error（漏抽开发支出附注）改进 section_router；针对 hallucination 改进 prompt 约束。
6. **页码稳定化**：在 MinerU 解析后做页码对齐处理。

## 15. 数据质量说明

- 所有数据来自巨潮资讯网（www.cninfo.com.cn）公开公告，采集时间为 2026 年 6 月。
- PDF 文件未经任何修改，SHA256 审计确认 150 个文件均为唯一 hash。
- 研发投入金额的单位从年报原始单位（元/万元/百万元）统一归一化为万元。
- `evidence_text` 为 LLM 从 MinerU 输出 Markdown 中提取的原文片段，不是 LLM 自由生成的文本。
- `capitalization_related`（问询标签）为 MVP 版本，基于 Tier-1 关键词剪枝 + LLM 语义二分类，标注为"非监管级判定"。
- 35 条人工评估样本的评估 agent 与 extractor 使用同源 LLM，存在系统共错风险，评估结论不代表独立审计水平。
- `page_no` 字段因 MinerU 输出不保留稳定页码，大部分为 0/null，这是已知局限而非实现 bug。
